# Spec for YAML-Driven Generic Node Executor

feature_slug: yaml-driven-generic-node-executor  
screen_comp (if available): N/A

## Intent and Desired Outcomes
- Business intent: Eliminate redundant per-flow TypeScript wrapper functions for common node patterns (ask a question, ingest an answer, display output, send greeting) by making the graph compiler auto-generate node handlers from YAML config. This reduces the amount of custom code required for each flow to only the genuinely domain-specific handlers.
- User problem: CFS has ~20 handler functions, but roughly 8-10 are thin wrappers over infrastructure primitives (`AskQuestion`, `AskWithRephrase`, `applyUserAnswer`, `ingestDispatcher`, `pushAI`). Each new flow would need to duplicate these trivial wrappers. Additionally, ingest handlers currently embed "what question to ask next" sequencing that belongs in YAML routing rules.
- Desired outcomes:
  - Nodes of kind `question` can be fully defined in YAML (config string key, question key, interpolation sources) with no TypeScript handler needed.
  - Nodes of kind `ingest` for simple field capture (name, role, industry, timeframe) use a generic handler driven by `ingestFieldMappings` + `applyUserAnswer`, with sequencing moved to `routingRules`.
  - A generic `initGreeting` handler sends config strings and sets initial session state, driven by YAML.
  - A generic `displayOutput` handler reads a state path and pushes it as an AI message, driven by YAML.
  - CFS thin-wrapper handlers are replaced by generic equivalents; domain-specific handlers remain as custom TypeScript.
  - All existing CFS tests pass with no behavioral regression.
- Success metrics (observable):
  - CFS `cfs-handlers.ts` handler count drops from ~16 to ~6 (only domain-specific custom handlers remain; ~10 replaced by generic YAML-driven equivalents).
  - Node files `step1-know-your-customer-nodes.ts` shrinks by at least 40% (generic ask/ingest handlers removed).
  - A new flow's simple ask/ingest nodes require zero TypeScript — only YAML config.
  - `npm run build` and `npm test` pass after each phase.

## Terminology
- **Generic handler** — Auto-generated by the compiler from YAML config (`questionConfig`, `greetingConfig`, `displayConfig`). No TypeScript needed per flow.
- **Custom handler** — Written in TypeScript, registered in the handler registry, referenced by `handlerRef` in the YAML. Required when the node has domain logic (service calls, AI invocations, conditional branching) that cannot be expressed declaratively.
- **Routing rules** — YAML-only declarations under `routingRules` that decide which node runs next based on state conditions. They control graph traversal, not node execution.

## Summary
The CFS node files contain many handler functions that are thin wrappers over existing infrastructure primitives (`AskQuestion`, `AskWithRephrase`, `applyUserAnswer`, `ingestDispatcher`). Functions like `nodeAskUserName`, `nodeAskIndustry`, `nodeAskRole`, `nodeAskTimeframe`, `nodeInit`, and `nodeDisplayReadout` follow identical patterns: read a config string, optionally interpolate state values, call a primitive. This spec defines a generic node executor built into the graph compiler that auto-generates handlers for common node patterns when the YAML provides enough config. CFS ingest handlers that embed sequencing logic ("after name, ask industry") will be simplified to just sanitize-and-store, with the sequencing moved entirely to YAML routing rules. The existing CFS flow is refactored to use the generic executor, proving it works before new flows adopt it.

## Scenario Coverage
### Primary Success Scenario
- Given CFS `flow.yaml` defines `askUserName` as `kind: question` with a new `questionConfig` block specifying `stringKey: "step1.nameQuestion"`, `questionKey: "S1_NAME"`
- When the graph compiles
- Then the compiler auto-generates a handler that calls `AskQuestion.run` with the config string and question key. No TypeScript handler is needed. The chatbot asks the user's name identically to today.

### Variation Scenarios
- Scenario A: A question node with `allowAIRephrase: true` and `rephraseContext` in YAML. The compiler generates a handler using `AskWithRephrase` instead of `AskQuestion`. CFS `nodeAskRole` and `nodeAskTimeframe` use this path.
- Scenario B: An ingest node with no custom handler, where `ingestFieldMappings` provides the target field and `routingRules` handles sequencing. The compiler generates a generic handler that calls `applyUserAnswer` and returns. CFS `handleName` and `handleTimeframe` use this path.
- Scenario C: A greeting node with `kind: question` and `greetingConfig` listing config string keys to send as initial messages. CFS `nodeInit` uses this path.
- Scenario D: A display node with `kind: compute` and `displayConfig` specifying a state path to read and format. CFS `nodeDisplayReadout` uses this path.
- Scenario E: A domain-specific node (e.g. `nodeDetermineUseCases`) still uses a custom `handlerRef`. The compiler falls back to the handler registry as it does today.

### Failure / Edge Scenarios
- Scenario E1: A YAML node has both `questionConfig` and a `handlerRef` that resolves in the registry -> The explicit `handlerRef` takes precedence; the generic config is ignored. Log a warning.
- Scenario E2: A YAML `questionConfig` references a `stringKey` that doesn't exist in `config.strings` -> `configString` returns the fallback default. No crash. Existing behavior.
- Scenario E3: An ingest node has no `ingestFieldMappings` entry for its question key and no `handlerRef` -> The generic handler calls `applyUserAnswer`, which falls through to its hardcoded fallback mappings (S1_NAME, S1_ROLE, etc.). Existing behavior preserved.

## Functional Requirements
- FR-1: The graph compiler should auto-generate a handler for `kind: question` nodes that have a `questionConfig` block in YAML and no resolvable `handlerRef` in the registry.
- FR-2: `questionConfig` should support: `stringKey` (config string key), `questionKey` (routing key), `interpolateFrom` (map of placeholder -> state path), `allowAIRephrase` (boolean), `rephraseContext` (inline or from state).
- FR-3: The graph compiler should auto-generate a handler for `kind: ingest` nodes that have no resolvable `handlerRef` and whose `questionKey` matches an entry in `ingestFieldMappings`.
- FR-4: Generic ingest handlers should call `applyUserAnswer` to sanitize and store the answer, then return. Sequencing ("what comes next") must be handled by `routingRules`, not the handler.
- FR-5: The graph compiler should support a `greetingConfig` on `kind: question` nodes that lists config string keys to push as sequential AI messages and sets initial session context.
- FR-6: The graph compiler should support a `displayConfig` on `kind: compute` nodes that reads a state path and pushes the result as an AI message.
- FR-7: When a node has both a `questionConfig`/`greetingConfig`/`displayConfig` and a `handlerRef` that resolves in the registry, the explicit `handlerRef` takes precedence.
- FR-8: The following CFS handlers should be replaced by generic YAML-driven equivalents (~10 total):
  - Question nodes: `nodeAskUserName`, `nodeAskIndustry`, `nodeAskRole` (with `allowAIRephrase`), `nodeAskTimeframe` (with `allowAIRephrase`)
  - Greeting node: `nodeInit`
  - Display node: `nodeDisplayReadout`
  - Ingest handlers: `handleName`, `handleTimeframe`, `handleConfirmStart`, `handleKycConfirm`
- FR-9: CFS ingest handlers that currently embed "ask next question" sequencing (`handleName` -> `nodeAskIndustry`, `handleIndustry` -> `nodeAskRole`) should be simplified to sanitize-and-store only. The "what comes next" logic should be in `routingRules`.
- FR-10: `handleConfirmStart` and `handleKycConfirm` have affirmative-check branching ("yes" -> proceed, "no" -> show rejection message). The generic ingest handler should support an `affirmativeCheckConfig` in YAML that specifies: `rejectStringKey` (config string for rejection message), `rejectPatch` (session context to set on rejection). Routing rules handle the "yes" branch.
- FR-11: All existing CFS tests should pass unchanged after the refactor.

## Possible Edge Cases
- `handleIndustry` has complex clarification logic (Firecrawl web search for sub-industries, LLM industry matching, multi-turn clarification with AI-generated suggestions). Cannot be generalized because it integrates external services and uses conditional AI reasoning.
- `handleRole` has persona resolution (LLM call) + role assessment agent invocation + unrecognized-role re-prompt. Cannot be generalized because it chains multiple sequential AI calls with branching based on AI output.
- `handleConfirmRole` re-invokes persona resolution when user rejects, manages a retry counter, orchestrates a multi-step correction flow. Cannot be generalized because it has stateful retry logic coupled with AI calls.
- Some ingest handlers call the next question node directly (e.g. `handleName` calls `nodeAskIndustry`). After refactor, `routingRules` must replicate this sequencing exactly.
- The `nodeInit` greeting pushes two messages with different content; `greetingConfig` must support an ordered list of string keys.
- `nodeAskIndustry` builds a multi-part question from three config strings; `questionConfig` must support either a single `stringKey` or a `stringKeys` array with join behavior.

## Acceptance Criteria
- AC-1: CFS `flow.yaml` defines `askUserName` with `questionConfig` and no `handlerRef`. The chatbot asks "What's your first name?" identically to today -> pass.
- AC-2: CFS `flow.yaml` defines `ingestUserName` with no `handlerRef`. The generic handler sanitizes the name via `applyUserAnswer` and returns. Routing rules direct to the next node -> pass if name is captured correctly.
- AC-3: CFS `flow.yaml` defines `sendIntroAndAskUseCaseGroup` with `greetingConfig`. The chatbot sends the greeting and follow-up messages identically to today -> pass.
- AC-4: CFS `flow.yaml` defines `nodeDisplayReadout` with `displayConfig`. The readout is displayed identically to today -> pass.
- AC-5: CFS `cfs-handlers.ts` registers only domain-specific custom handlers (~6). Generic nodes have no handler registration -> pass.
- AC-6: CFS `routingRules.routeInitFlow` includes rules for the sequencing previously embedded in `handleName`, `handleTimeframe`, etc. -> pass if conversation flow is identical.
- AC-7: All existing CFS tests pass -> `npm test` exit code 0.
- AC-8: `npm run build` completes with no new TypeScript errors -> pass.
- AC-9: A new flow can define simple ask/ingest nodes in YAML with zero TypeScript handlers -> pass by code inspection.

## Dependencies and Constraints
- Dependencies:
  - This spec should be executed **before** `multi-flow-handler-architecture`. Reducing handler count first simplifies the multi-flow reorganization. The generic executor does not require multi-flow infrastructure; it works within the current CFS-only architecture.
  - `ingestFieldMappings` already exists in `flow.yaml` and `applyUserAnswer` already reads it.
  - `configString` + `interpolate` infrastructure already exists.
  - `AskQuestion`, `AskWithRephrase` primitives already exist.
- Constraints:
  - Must not change behavior of any existing CFS conversation flow.
  - Must not change the GraphDSL Zod schema in a backward-incompatible way (new optional fields only).
  - CFS `routingRules.routeInitFlow` will grow to include the sequencing currently in ingest handlers; this is acceptable.
  - The `readout_context` state path used by `displayConfig` is CFS-specific but the mechanism is generic.

## Handlers That Remain Custom (Post-Refactor)
The following ~6 handlers contain domain-specific business logic that cannot be expressed declaratively in YAML:

1. **`handleIndustry`** — Calls Firecrawl for sub-industry discovery, invokes LLM for industry match, triggers multi-turn clarification loop with AI-generated suggestions. Has external service integration + conditional AI reasoning.
2. **`handleRole`** — Invokes LLM persona resolution, triggers role assessment AI agent, handles unrecognized role re-prompt. Has multiple sequential AI calls and conditional branching based on AI output.
3. **`handleConfirmRole`** — Re-invokes persona resolution when user rejects, manages retry counter, orchestrates multi-step correction flow. Has stateful retry logic coupled with AI calls.
4. **`nodeDetermineUseCases`** — Orchestrates use-case discovery via AI agents, manages multi-turn narrowing flow. Domain-specific orchestration.
5. **`nodeDeterminePillars`** / **`nodeBuildReadout`** — AI-driven analysis and readout generation. Domain-specific orchestration.
6. **`nodeKnowYourCustomerEcho`** — KYC summary echo with AI formatting. Domain-specific.

These are referred to as **custom handlers** — they are TypeScript functions registered in the handler registry and referenced by `handlerRef` in the YAML.

## Out of Scope
- Multi-flow support (covered by `multi-flow-handler-architecture` spec).
- Generalizing the custom handlers listed above.
- Changes to the state schema (`CfsStateSchema`).
- New flow creation — only infrastructure + CFS refactor.
- UI template changes.

## Open Questions
- Q1: Should `questionConfig` be a new top-level field on the node definition in the Zod schema, or nested under a generic `nodeConfig` object? A top-level field is simpler; `nodeConfig` is more extensible.
- Q2: For `nodeAskIndustry` which assembles a question from 3 config strings (`industryIntro`, `industryBody`, `industryPrompt`), should `questionConfig.stringKeys` be an array that joins with `\n\n`, or should this remain a custom handler? It's borderline — the assembly logic is trivial but specific.
- Q3: Should the compiler log a warning when a node has both `questionConfig` and a resolvable `handlerRef`, or silently prefer the `handlerRef`?
